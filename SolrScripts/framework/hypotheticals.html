  • Code Review ID: 87499
  • Feature: What if
  • Number of comment: 4
  • Comment:
    Patch Set 2: To stir the pot a bit, what if we reduce the number of current app domains in our policy. In certain cases there doesn't seem to be a real distinction that needs to be drawn among certain app domains. I believe there remains a need for the named uid app domains (bluetooth, nfc, radio, system_app) given that those apps typically need access to security critical pieces of the system, i.e the radio app needs access to the radio socket. The platform_app domain can likely stay as well given that it represents the platform cert on the phone. These apps typically need access to certain areas of the file system such as asec and app install directories which are critical to the security ofthe device. Plus theses apps are usually the ones referred to one when talking about signature middleware permissions which open up other access vectors that might need to be granted. But as it stands I'm not seeing a real need to keep separate domains for shared, release, and media apps. Seems like we could coalesce these in some way. Would it make sense for these to be untrusted? Lastly, we can use this script to label the apps under /system/priv-app. This seems to be needed given that apps under /system/priv-app typically are using a system middleware permission which grants certain access rights not afforded to other apps (i.e access to /cache). Plus, a lot of apps under /system/priv-app are Gapps that are causing denials because they are being labeled as untrusted. For those keeping score: (named uid app domains), (platform signed apps domain), (apps in /system/priv-app), (other apps under /system/app ?= untrusted),(third party) This labeling model would seem to reinforce the current model that Android is using for apps. Thoughts?
