    Fair question. After some research, I think that's not actually possible to do this. That is: I don't think we'll ever know the reason for a short write. In detail: - On a write() syscall, the kernel returns a ssize_t to userspace. That ssize_t can _either_ report a number of bytes written (with a non-negative number), _or_ the ssize_t can report an error condition (with a negative number). - It is because the kernel can only return one of these values, that Torvalds and Morton argue about whether a partial write() should return the number of bytes written, or the error which caused a short write. - Torvalds concludes that the kernel should return the number of bytes written. - Hence, in the case of a partial write, we don't actually know the error code. One thing that might be confusing is that userspace has two concepts (return value and errno), while the kernel only has one (ssize_t with non-negative values for success, and negative values for error). The bridge between these two worlds is libc (in our case, bionic). The C library reads the return value of the syscall, and maps that accordingly. See, in particular, [1] and [2]. While that explains what data we can get, the situation seems unsatisfactory. As you point out, how do we know why the write was truncated? Torvalds' position is that you just retry with the unwritten data. He doesn't paint the picture in full detail, but here's what I think he has in mind... The retry has three possible outcomes: a) The retry succeeds fully. In this case: we don't care why the earlier write was truncated. Maybe the socket buffer was full, but that kind of transient condition is totally normal. b) The next write fails completely. In this case, the kernel returns the error code, bionic returns -1 and sets errno, and we report the error via the code at line 70. c) This new retry is also partially successful. In that case, repeat until we hit case a) or b).   • Code Review ID: 80871
  • Feature: prospect
  • Comment:
    If there are multiple certs that sign an app and multiple mac_permissions.xml signature entries that match then it's the first signature that is returned by the call to pkg.mSignatures that hits. Since the mSignatures is just an array, the first signature would be whatever key is pulled off the app first and stored by the PMS. We always approached this multiple sig problem in a number of ways. One is that apks are rarely signed with multiple certs even though it is possible. I've found a few apps on the market that have this feature, the mint financial app was one, but not many. Two, is that we believe that if an app is signed with multiple certs, and there are stanzas for each signature, then there is already a level of trust among those stanzas that the policy creator is ok with. Meaning, if the first stanza hit but not the second, or vice versa, it was fine. Beyond that, we also didn't want to force OEMs to know a priori all possible certs that could ever be used to double sign an app they cared about. This allowed a base cert that could be used, and represented in the mac_permissions.xml file, without having to either create stanzas for all possible combinations or be forced to distribute a new policy file just because of a new cert that was used to double sign an app. We thought the prospect of matching on all certs to not scale well in certain cases and was more of a motivating factor than the sharp restriction that all certs match.
